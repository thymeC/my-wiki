# 知识库目录与组织模型参考

> **用途**：把常见 **PKM / 笔记目录** 方法论整理在一处，方便查阅与对比。  
> **本仓库实际怎么摆文件夹** 以根目录 **[[REPO_LAYOUT]]** 为准（勿与本页通识混为一谈）。  
> **主题 MOC 入口**：[[moc_index]]。

---

## 一、核心原则（选任何模型前）

- **先行动，再完美**：从简单结构起步，使用中迭代。  
- **可查找 > 分类整齐**：强搜索 + 链接，往往胜过深层文件夹。  
- **链接可补层级**：网状连接（`[[笔记]]`）让一篇笔记可属于多个主题，而不必复制文件。  
- **知识库为用不为囤**：定期回顾、搬迁、删除。

---

## 二、总览表（按主轴选模型）

| 模型 | 主轴 | 一句话 | 更适配的工具倾向 |
|------|------|--------|-------------------|
| **PARA** | 可操作性 | 按「做不做、是否结束」分 Projects / Areas / Resources / Archives（常加 Inbox） | Notion、Obsidian、任意文件夹 |
| **MOC** | 主题导航 | 用「地图笔记 + 链接」组织话题，弱化深文件夹 | Obsidian、Logseq、Roam |
| **Zettelkasten** | 原子化 + 关联 | 小笔记、强链接、常配固定 ID；目录往往很扁 | The Archive、ZK 插件、Logseq |
| **Johnny.Decimal** | 可预测编号 | 有限层级 + 十进制编号，路径稳定、可口述 | 纯文本、任意文件系统 |
| **杜威式 / 人生分区** | 层级类别 | 工作/学习/生活等大区 + 子类；可数字前缀排序 | 文件夹党、多设备同步 |
| **LATCH** | 元维度 | 组织方式不外乎：位置、字母、时间、类别、层级 | 设计结构前的自检 |
| **时间轴 / Daily** | 发生时间 | 以日期页为入口，再链到主题 | Logseq、Obsidian Daily |
| **扁平 + 标签 + 搜索** | 检索 | 几乎不设文件夹，靠命名与搜索 | 笔记量中等、搜索强 |
| **Evergreen** | 写作迭代 | 反复改写、标题即主张、高质量链接；目录次要 | Obsidian、公开站点 |
| **GTD** | 工作流 | 收集→理清→组织→回顾；常驱动 Inbox 与项目节奏 | 与 PARA、收件箱习惯合拍 |

---

## 三、PARA（Projects · Areas · Resources · Archives）

**核心**：按信息的 **可操作性** 与 **当前价值** 分类，而非按学科树。

**典型结构（教程里常见带编号前缀）**：

- `Inbox`：新想法、剪藏入口  
- `Projects`：有明确截止的短期任务  
- `Areas`：需长期维护的责任范围（健康、职业、财务等）  
- `Resources`：有兴趣的主题资料  
- `Archives`：已结束或不再活跃的内容  

**优点**：和行动结合紧；**缺点**：要维护活跃/归档状态，Projects 易堆。

**与本仓库**：见 [[REPO_LAYOUT]]（本库用 `moc`+`resources` 分担 Inbox 角色等，并非教科书级 PARA）。

---

## 四、MOC（Map of Content / 内容地图）

**核心**：用 **专门笔记** 做主题索引，列出 `[[相关笔记]]`，可嵌套子主题。

**做法示例**：文件夹只保留少数层（如 `_MOCs`、`Notes`），每个大主题一张 MOC。

**优点**：灵活、一篇可多主题归属；**缺点**：完全不要文件夹时，部分人会缺乏安全感——可与 PARA **并联**（文件夹一个家，MOC 多扇窗）。

**与本仓库**：探索期地图放在 `moc/`，见 [[REPO_LAYOUT]]。

---

## 五、杜威十进制式 + LATCH

**LATCH**（Richard Saul Wurman）：**L**ocation、**A**lphabet、**T**ime、**C**ategory、**H**ierarchy —— 检查你当前结构主要靠哪一维。

**杜威式组合示例**：

- 顶层按 **生活/工作/学习** 等板块  
- 二级按 **领域** 再分  
- 文件名可 **日期 + 主题**，如 `2026-05-05_topic.md`  
- 常用 **`10_`、`20_` 前缀** 固定排序  

**优点**：直观、稳定、易备份；**缺点**：强交叉主题时归类会纠结（可用 MOC / 链接缓解）。

**与本仓库**：`areas/` 下英文主题树 + 可选数字前缀，即类似思路；细则仍看 [[REPO_LAYOUT]]。

---

## 六、Zettelkasten（卡片盒）

**核心**：一卡一事（尽量小）、强制链接；常配 **固定 ID**（日期或字母数字）稳定引用。

**适合**：研究型写作、论点链条；**代价**：剪藏需消化成自己的话，否则易成坟场。

**与本仓库**：可只借「原子化 + 链接」，不必全盘 ZK；成稿落在 `areas/`。

---

## 七、Johnny.Decimal

**核心**：严控层级深度，用 **十进制编号**（如 `12.03`）表示「领域.类别」，路径短、可排序。

**适合**：讨厌随意改名、要路径可预测；**代价**：规则要守；强交叉仍需链接。

---

## 八、时间轴优先（Daily Notes）

**核心**：以 **日期页** 为默认入口，碎片进当日，再链到主题或 MOC。

**适合**：日记、会议、灵感多；**代价**：按主题回顾依赖反向链接或 MOC。

**与本仓库**：可与 [[REPO_LAYOUT]] 四目录并行，例如另设 `journal/` 或在 Obsidian 里配 Daily 目录。

---

## 九、扁平 + 标签 + 搜索

**核心**：极少子文件夹，依赖命名、标签、全文搜索。

**适合**：笔记量可控、搜索极强；**代价**：量大后靠标签一致性变难。

---

## 十、Evergreen notes（常青笔记）

**核心**：持续改写、标题像主张、笔记间 **高质量链接**；文件夹次要。

**适合**：长期打磨的精品文、对外输出；**代价**：对纯剪藏型用户纪律要求高。

---

## 十一、GTD 与收件箱

**核心**：**收集 → 理清 → 组织 → 回顾**；Inbox 必须定期清空。

**与本仓库**：定期清 `resources/`、把 `moc/` 成熟块迁 `areas/`，精神接近 GTD；项目对齐 `projects/`。

---

## 十二、怎么混搭（实用建议）

1. **顶层文件夹一种哲学即可**，避免两套顶层逻辑互抢。  
2. **交叉主题交给 MOC + 链接**，不复制文件。  
3. **需要时间维度再加 Daily**，与 PARA 不互斥。  
4. **ZK / Evergreen 可只取子集**（原子化、自己的话、强链接）。  
5. **任何模型都败给「只收不整理」** —— 定周/双周最小回顾。

---

## 十三、工具与起步结构（简要）

| 需求倾向 | 常见选择 |
|----------|----------|
| 文件在手 + 隐私 | **Obsidian**（常配 MOC） |
| 数据库 + 协作 | **Notion**（常配 PARA） |
| 大纲 + 碎片日志 | **Logseq**（自然 MOC / Daily） |
| 简单跨平台文件夹 | **OneNote** + 杜威式目录 |

**最低阻力起步示例**：

```
./MyNotes/
├── 0_Inbox/
├── 1_Active/        # 当前项目与学习
├── 2_Area/          # 长期关注
├── 3_Archive/
└── _MOC/            # 可选：索引笔记
```

**选型口诀**：

- 偏 **行动与产出** → PARA + Notion/Obsidian  
- 偏 **关联与思考** → MOC + Obsidian/Logseq  
- 偏 **传统稳定** → 杜威式 + 任意文件系统
